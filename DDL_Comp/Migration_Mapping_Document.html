1. Executive Summary

Migration Objective

Migrate customer, entity, and document data from the Zenith legacy system to a modern cloud-based platform with enhanced data governance, audit capabilities, and multi-tenancy support.

Scope

  • Source System: Zenith Legacy Database (SQL Server)
  • Target System: Modern Cloud Platform (SQL Server)
  • Migration Type: One-time bulk migration with potential for incremental updates
  • Data Volume: To be determined through profiling

Key Migration Risks

Critical Risks

  • Key Resolution Complexity: Source uses composite natural keys (CUSTOMER + CLIENT_GROUP / CLIENT), target uses surrogate GUIDs requiring comprehensive crosswalk strategy
  • Missing Reference Data: Target requires EntityType, CustomerType, BusinessParty, ChartOfAccountsName - no direct source equivalents identified
  • PII/Compliance Data: SSN, FEDERAL_ID, EIN fields require encryption/masking strategy
  • Audit Trail Gap: Source has minimal audit fields; target requires comprehensive audit metadata

Medium Risks

  • Data Type Conversions: CHAR to NVARCHAR, datetime to datetime2, INT to BIGINT for document sizes
  • Null Handling: Source allows NULLs in many fields; target has stricter NOT NULL constraints
  • Business Logic Assumptions: CLIENT_GROUP to Enterprise mapping logic needs SME validation
  • Document Encoding: Source uses CHAR encoding; target uses NVARCHAR requiring encoding validation

Key Assumptions

  • Each source CUSTOMER record maps to one target Customer record
  • CLIENT_GROUP represents an Enterprise-level grouping concept
  • CLIENT records map to Entity records within an Enterprise
  • DBFILE and DBFILE_DETAIL map to Document and DocumentDetail respectively
  • All migrated records will use a migration service account for CreatedBy/ModifiedBy initially
  • IsActive flag defaults to TRUE (1) for all migrated records unless business rules dictate otherwise
  • Document sizes in DBFILE.KILOBYTES need conversion to bytes for DocumentSize (BIGINT)
  • CUSTOMER_SECURITY table is not directly migrated but may inform access control setup

11. Crosswalk and Key Management Plan

Crosswalk Table Design

CREATE TABLE Migration_Key_Map ( MigrationKeyMapId UNIQUEIDENTIFIER PRIMARY KEY DEFAULT NEWID(), SourceSystem VARCHAR(50) NOT NULL, SourceTable VARCHAR(100) NOT NULL, SourceNaturalKey VARCHAR(500) NOT NULL, SourceContext VARCHAR(100) NULL, TargetTable VARCHAR(100) NOT NULL, TargetId UNIQUEIDENTIFIER NOT NULL, LoadBatchId UNIQUEIDENTIFIER NOT NULL, CreatedDate DATETIME2 NOT NULL DEFAULT GETDATE(), CONSTRAINT UQ_Migration_Key_Map UNIQUE (SourceSystem, SourceTable, SourceNaturalKey, SourceContext, TargetTable) ); CREATE INDEX IX_Migration_Key_Map_Source ON Migration_Key_Map(SourceSystem, SourceTable, SourceNaturalKey); CREATE INDEX IX_Migration_Key_Map_Target ON Migration_Key_Map(TargetTable, TargetId);

Source Natural Keys

Source Table Natural Key Target Table Target Surrogate Key Context Required?
CUSTOMER CUSTOMER Customer CustomerId No
CUSTOMER CUSTOMER + CLIENT_GROUP Enterprise Id No
DBFILE CUSTOMER + CLIENT Entity Id No
DBFILE CUSTOMER + CLIENT + DBFILE_RESOURCE + DBFILE_PATH + DBFILE_NAME + VERSION Document DocumentId No
DBFILE_DETAIL CUSTOMER + DBFILE_ID + DBFILE_LINE DocumentDetail DocumentDetailId No

Deterministic Key Generation Rules

Customer Key Generation

-- Generate deterministic CustomerId from CUSTOMER natural key DECLARE @SourceKey VARCHAR(500) = RTRIM(CUSTOMER); DECLARE @CustomerId UNIQUEIDENTIFIER = CAST(HASHBYTES('MD5', CONCAT('CUSTOMER|', @SourceKey)) AS UNIQUEIDENTIFIER);

Enterprise Key Generation

-- Generate deterministic Enterprise Id from CUSTOMER + CLIENT_GROUP DECLARE @SourceKey VARCHAR(500) = CONCAT(RTRIM(CUSTOMER), '|', RTRIM(CLIENT_GROUP)); DECLARE @EnterpriseId UNIQUEIDENTIFIER = CAST(HASHBYTES('MD5', CONCAT('ENTERPRISE|', @SourceKey)) AS UNIQUEIDENTIFIER);

Entity Key Generation

-- Generate deterministic Entity Id from CUSTOMER + CLIENT DECLARE @SourceKey VARCHAR(500) = CONCAT(RTRIM(CUSTOMER), '|', RTRIM(CLIENT)); DECLARE @EntityId UNIQUEIDENTIFIER = CAST(HASHBYTES('MD5', CONCAT('ENTITY|', @SourceKey)) AS UNIQUEIDENTIFIER);

Document Key Generation

-- Generate deterministic DocumentId from composite natural key DECLARE @SourceKey VARCHAR(500) = CONCAT( RTRIM(CUSTOMER), '|', RTRIM(CLIENT), '|', RTRIM(DBFILE_RESOURCE), '|', RTRIM(DBFILE_PATH), '|', RTRIM(DBFILE_NAME), '|', CAST(VERSION AS VARCHAR(10)) ); DECLARE @DocumentId UNIQUEIDENTIFIER = CAST(HASHBYTES('MD5', CONCAT('DOCUMENT|', @SourceKey)) AS UNIQUEIDENTIFIER);

Crosswalk Population Strategy

Load Group 1: Crosswalk Preparation

  1. Customer Crosswalk: Generate and store CustomerId mappings
  2. Enterprise Crosswalk: Generate and store Enterprise Id mappings
  3. Entity Crosswalk: Generate and store Entity Id mappings
  4. Document Crosswalk: Generate and store DocumentId mappings
-- Example: Populate Customer crosswalk INSERT INTO Migration_Key_Map (SourceSystem, SourceTable, SourceNaturalKey, SourceContext, TargetTable, TargetId, LoadBatchId) SELECT 'Zenith' AS SourceSystem, 'CUSTOMER' AS SourceTable, RTRIM(CUSTOMER) AS SourceNaturalKey, NULL AS SourceContext, 'Customer' AS TargetTable, CAST(HASHBYTES('MD5', CONCAT('CUSTOMER|', RTRIM(CUSTOMER))) AS UNIQUEIDENTIFIER) AS TargetId, @BatchId AS LoadBatchId FROM ZenithTestDDL.CUSTOMER;

FK Resolution Logic

Resolve CustomerId FK

-- Resolve CustomerId from CUSTOMER natural key SELECT m.TargetId AS CustomerId FROM Migration_Key_Map m WHERE m.SourceSystem = 'Zenith' AND m.SourceTable = 'CUSTOMER' AND m.SourceNaturalKey = RTRIM(@SourceCustomer) AND m.TargetTable = 'Customer';

Resolve EnterpriseId FK

-- Resolve EnterpriseId from CUSTOMER + CLIENT_GROUP SELECT m.TargetId AS EnterpriseId FROM Migration_Key_Map m WHERE m.SourceSystem = 'Zenith' AND m.SourceTable = 'CUSTOMER' AND m.SourceNaturalKey = CONCAT(RTRIM(@SourceCustomer), '|', RTRIM(@SourceClientGroup)) AND m.TargetTable = 'Enterprise';

Resolve EntityId FK

-- Resolve EntityId from CUSTOMER + CLIENT SELECT m.TargetId AS EntityId FROM Migration_Key_Map m WHERE m.SourceSystem = 'Zenith' AND m.SourceTable = 'DBFILE' AND m.SourceNaturalKey = CONCAT(RTRIM(@SourceCustomer), '|', RTRIM(@SourceClient)) AND m.TargetTable = 'Entity';

Re-run Handling

  • Idempotent Key Generation: Deterministic GUID generation ensures the same source record always produces the same target GUID
  • Crosswalk Upsert: Use MERGE or INSERT...ON DUPLICATE KEY UPDATE to handle re-runs
  • Batch Tracking: LoadBatchId tracks which batch created each mapping
  • Duplicate Prevention: Unique constraint on (SourceSystem, SourceTable, SourceNaturalKey, SourceContext, TargetTable) prevents duplicates

Collision Prevention

  • Prefix Strategy: Use table-specific prefixes in HASHBYTES input (e.g., 'CUSTOMER|', 'ENTERPRISE|')
  • Context Field: Use SourceContext to differentiate when same natural key feeds multiple targets
  • Composite Keys: Include all components of composite natural keys in hash input

15. Audit Log Design

Migration Audit Log Table

CREATE TABLE Migration_Audit_Log ( AuditLogId UNIQUEIDENTIFIER PRIMARY KEY DEFAULT NEWID(), BatchId UNIQUEIDENTIFIER NOT NULL, SourceSystem VARCHAR(50) NOT NULL, SourceTable VARCHAR(100) NOT NULL, TargetTable VARCHAR(100) NOT NULL, LoadGroup INT NOT NULL, RecordsRead BIGINT NOT NULL DEFAULT 0, RecordsInserted BIGINT NOT NULL DEFAULT 0, RecordsUpdated BIGINT NOT NULL DEFAULT 0, RecordsRejected BIGINT NOT NULL DEFAULT 0, RecordsSkipped BIGINT NOT NULL DEFAULT 0, StartTime DATETIME2 NOT NULL, EndTime DATETIME2 NULL, DurationSeconds AS DATEDIFF(SECOND, StartTime, EndTime), Status VARCHAR(20) NOT NULL CHECK (Status IN ('Running', 'Completed', 'Failed', 'Partial')), ErrorMessage NVARCHAR(MAX) NULL, ExecutedBy UNIQUEIDENTIFIER NOT NULL, ServerName VARCHAR(100) NULL, DatabaseName VARCHAR(100) NULL, CreatedDate DATETIME2 NOT NULL DEFAULT GETDATE() ); CREATE INDEX IX_Migration_Audit_Log_Batch ON Migration_Audit_Log(BatchId); CREATE INDEX IX_Migration_Audit_Log_Status ON Migration_Audit_Log(Status); CREATE INDEX IX_Migration_Audit_Log_Target ON Migration_Audit_Log(TargetTable);

Audit Log Usage

  • BatchId: Groups all migration activities for a single execution run
  • LoadGroup: Tracks which dependency group the load belongs to
  • RecordsRead: Count of source records processed
  • RecordsInserted: Count of new target records created
  • RecordsUpdated: Count of existing target records updated (for re-runs)
  • RecordsRejected: Count of records that failed validation
  • RecordsSkipped: Count of records intentionally skipped (e.g., already migrated)
  • Status: Running | Completed | Failed | Partial
  • ErrorMessage: Detailed error information if status is Failed

Sample Audit Log Insert

-- Start of migration step DECLARE @AuditLogId UNIQUEIDENTIFIER = NEWID(); DECLARE @StartTime DATETIME2 = GETDATE(); INSERT INTO Migration_Audit_Log (AuditLogId, BatchId, SourceSystem, SourceTable, TargetTable, LoadGroup, StartTime, Status, ExecutedBy) VALUES (@AuditLogId, @BatchId, 'Zenith', 'CUSTOMER', 'Customer', 2, @StartTime, 'Running', @MigrationUserId); -- Perform migration... -- End of migration step UPDATE Migration_Audit_Log SET EndTime = GETDATE(), Status = 'Completed', RecordsRead = @SourceCount, RecordsInserted = @InsertCount, RecordsRejected = @RejectCount WHERE AuditLogId = @AuditLogId;

Audit Log Reporting Queries

Migration Summary by Batch

SELECT BatchId, COUNT(*) AS TotalSteps, SUM(RecordsRead) AS TotalRecordsRead, SUM(RecordsInserted) AS TotalRecordsInserted, SUM(RecordsRejected) AS TotalRecordsRejected, MIN(StartTime) AS MigrationStartTime, MAX(EndTime) AS MigrationEndTime, SUM(DATEDIFF(SECOND, StartTime, EndTime)) AS TotalDurationSeconds FROM Migration_Audit_Log WHERE BatchId = @BatchId GROUP BY BatchId;

Failed Migration Steps

SELECT BatchId, SourceTable, TargetTable, StartTime, EndTime, ErrorMessage FROM Migration_Audit_Log WHERE Status = 'Failed' ORDER BY StartTime DESC;

Rejection Rate by Table

SELECT TargetTable, SUM(RecordsRead) AS TotalRead, SUM(RecordsRejected) AS TotalRejected, CAST(SUM(RecordsRejected) AS FLOAT) / NULLIF(SUM(RecordsRead), 0) * 100 AS RejectionRatePercent FROM Migration_Audit_Log WHERE BatchId = @BatchId GROUP BY TargetTable ORDER BY RejectionRatePercent DESC;

Retry Strategy

  • Transient Errors: Retry up to 3 times with exponential backoff (database connectivity, deadlocks)
  • Data Errors: Do not retry - log to reject table and continue
  • FK Lookup Failures: Check if parent record is in reject log; if so, cascade reject

Partial Load Handling

  • Transaction Scope: Use batch-level transactions (e.g., 1000 records per transaction)
  • Checkpoint Strategy: Commit after each successful batch, log progress
  • Resume Capability: Track last successfully processed record to enable resume from failure point

17. Migration Execution Guide

Pre-Migration Phase

Step 1: Profile Source Data

  • Run row count queries on all source tables
  • Identify NULL percentages for key fields
  • Validate natural key uniqueness
  • Check for data quality issues (invalid dates, negative amounts, etc.)
  • Analyze character encoding and special characters
  • Profile amount fields to confirm scale (dollars vs. cents)

Step 2: Confirm Source Natural Keys

  • Validate CUSTOMER is unique in CUSTOMER table
  • Validate CUSTOMER + CLIENT_GROUP is unique in CUSTOMER table
  • Validate CUSTOMER + CLIENT is unique in DBFILE table
  • Validate composite key uniqueness in DBFILE table
  • Validate CUSTOMER + DBFILE_ID + DBFILE_LINE is unique in DBFILE_DETAIL table

Step 3: Confirm Source Filter Criteria

  • Determine if any CUSTOMER records should be excluded (e.g., test customers)
  • Determine if any DBFILE records should be excluded (e.g., archived documents)
  • Document exclusion criteria and obtain SME approval

Step 4: Confirm Amount Scale and Date Semantics

  • Profile KILOBYTES field to confirm it represents kilobytes (not bytes)
  • Validate date fields are in expected format and range
  • Check for future dates or dates before system inception

Migration Phase

Step 5: Load Reference/Static Tables (Load Group 0)

  1. Seed EntityType table with business-approved entity types
  2. Seed CustomerType table with business-approved customer types
  3. Seed Currency table with supported currencies (if applicable)
  4. Validate reference data completeness

Step 6: Generate Deterministic Target Keys (Load Group 1)

  1. Generate CustomerId mappings and populate Migration_Key_Map
  2. Generate Enterprise Id mappings and populate Migration_Key_Map
  3. Generate Entity Id mappings and populate Migration_Key_Map
  4. Generate DocumentId mappings and populate Migration_Key_Map
  5. Validate crosswalk completeness (all source records have mappings)

Step 7: Populate Crosswalk Table (Load Group 1)

  • Crosswalk population is part of Step 6
  • Validate no duplicate mappings
  • Validate deterministic key generation (re-run produces same GUIDs)

Step 8: Load Customer (Load Group 2)

  1. Extract data from CUSTOMER table
  2. Resolve CustomerTypeId (use default or business rule)
  3. Transform and load into Customer table
  4. Log results to Migration_Audit_Log
  5. Validate row count matches source
  6. Validate no NULL CustomerName values

Step 9: Load Enterprise (Load Group 3)

  1. Extract distinct CUSTOMER + CLIENT_GROUP from CUSTOMER table
  2. Resolve CustomerId via crosswalk
  3. Transform and load into Enterprise table
  4. Log results to Migration_Audit_Log
  5. Validate row count matches distinct source combinations
  6. Validate all CustomerId FKs resolve

Step 10: Load Entity (Load Group 4)

  1. Extract distinct CUSTOMER + CLIENT from DBFILE table
  2. Join with CUSTOMER table for additional attributes
  3. Resolve EnterpriseId and CustomerId via crosswalk
  4. Resolve EntityTypeId (use default or business rule)
  5. Transform and load into Entity table
  6. Log results to Migration_Audit_Log
  7. Validate row count matches distinct source combinations
  8. Validate all FK relationships

Step 11: Load Document (Load Group 5)

  1. Extract data from DBFILE table
  2. Resolve CustomerId and EntityId via crosswalk
  3. Convert KILOBYTES to bytes
  4. Transform and load into Document table
  5. Log results to Migration_Audit_Log
  6. Validate row count matches source
  7. Validate all FK relationships
  8. Validate DocumentSize conversion

Step 12: Load DocumentDetail (Load Group 6)

  1. Extract data from DBFILE_DETAIL table
  2. Resolve DocumentId via crosswalk
  3. Convert CHAR to NVARCHAR
  4. Transform and load into DocumentDetail table
  5. Log results to Migration_Audit_Log
  6. Validate row count matches source
  7. Validate all DocumentId FKs resolve

Step 13: Load EntityGroupMapping (Load Group 7) - Optional

  1. If business has defined entity grouping logic, populate EntityGroupMapping
  2. Resolve EntityId and EntityGroupId
  3. Load mapping records
  4. Log results to Migration_Audit_Log

Post-Migration Phase

Step 14: Run Validation SQL

  1. Execute all validation queries from Validation Plan
  2. Document validation results
  3. Investigate and resolve any validation failures

Step 15: Review Rejects

  1. Query Migration_Reject_Log for all rejected records
  2. Categorize rejects by error type
  3. Determine if rejects are acceptable or require remediation
  4. Document reject analysis and resolution plan

Step 16: Reconcile Counts and Totals

  1. Compare source and target row counts
  2. Reconcile control totals (e.g., total document size)
  3. Investigate and explain any discrepancies
  4. Document reconciliation results

Step 17: Resolve SME Decisions

  1. Review SME Decision Tracker
  2. Obtain business approval for all SME Required items
  3. Update migration logic based on SME decisions
  4. Re-run affected migration steps if necessary

Step 18: Obtain Business Sign-Off

  1. Prepare migration summary report
  2. Present validation results to business stakeholders
  3. Obtain formal sign-off on migration results
  4. Document sign-off in project records

Step 19: Freeze Migration Batch

  1. Mark migration batch as final in Migration_Audit_Log
  2. Archive migration logs and reject logs
  3. Document final migration statistics

Step 20: Prepare Cutover

  1. Develop cutover plan and timeline
  2. Identify cutover dependencies and risks
  3. Prepare rollback plan
  4. Conduct cutover readiness review
  5. Execute cutover according to plan

Post-Cutover

  • Monitor system performance and stability
  • Implement EntityHistory audit mechanism
  • Conduct post-cutover validation
  • Address any post-cutover issues
  • Document lessons learned

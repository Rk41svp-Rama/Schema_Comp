Database Migration Package: AcquiredCompany to CoreCompany

1. Executive Summary

Migration Objective: To migrate customer and entity data from the legacy `AcquiredCompany` database schema to the new `CoreCompany` strategic platform. This migration is critical for consolidating customer relationship management and financial entity tracking into a single, modern system.

Scope: This document covers the schema analysis, mapping, and migration strategy for the `Customer` and `Entity` domains.

Source System: AcquiredCompany (SQL Server)

Target System: CoreCompany (SQL Server)

Total Tables Analyzed: 5 (Source), 6 (Target)

Total Mappings Generated: 25+

Mapping Confidence Summary:

  • High / Direct: 5
  • Medium / Derived / Lookup: 15
  • Low / Not Available: 3
  • SME Required: 8

Key Migration Risks

  • Data Quality in Source: The source schema uses `char` for most fields, which may contain inconsistent spacing and formatting. Extensive use of `TRIM` and data cleansing rules will be necessary.
  • Natural Key Ambiguity: The source `CUSTOMER` table has a composite primary key (`CUSTOMER`, `CLIENT_GROUP`). The business meaning of `CLIENT_GROUP` is not clear and requires SME input to ensure correct grouping into the target `Enterprise` or `Customer` objects.
  • Missing Foreign Keys: The source DDL lacks declared foreign key constraints, increasing the risk of orphaned records. Validation SQL must be robust.

Key Assumptions

  • The source `CUSTOMER` table represents the primary business relationship and will be the main source for the target `Customer` table.
  • The combination of `CUSTOMER` and `CLIENT_GROUP` from the source `CUSTOMER` table uniquely identifies a business entity that can be mapped to a target `Entity`.
  • All target `uniqueidentifier` keys will be newly generated and managed via a crosswalk table.

Key SME Decisions Required

  • Clarification of the business purpose of `CLIENT_GROUP` in the `CUSTOMER` table.
  • Default values for mandatory target fields like `IsActive`, `CustomerTypeId`, and `OfficeCode`.
  • Transformation logic for constructing `EntityAddress` from multiple source address fields.

3. Transformation Rules Library

  • String Trimming: `TRIM(source_column)` must be applied to all `char` source fields.
  • Surrogate Key Generation: `NEWID()` for `uniqueidentifier` fields.
  • Lookup Resolution: All FKs must be resolved using the `Migration_Key_Map` table. `SELECT TargetId FROM Migration_Key_Map WHERE SourceSystem = 'AcquiredCompany' AND SourceTable = '...' AND SourceNaturalKey = '...'`.
  • Audit Field Defaults: `CreatedBy` set to a static migration user ID. `CreatedDate` set to `GETUTCDATE()`.

4. Crosswalk and Key Management Plan

A central crosswalk table is mandatory for this migration to handle the shift from composite natural keys to surrogate `uniqueidentifier` keys.

Crosswalk Table Design

    CREATE TABLE Migration_Key_Map (
        SourceSystem VARCHAR(50) NOT NULL,
        SourceTable VARCHAR(100) NOT NULL,
        SourceNaturalKey VARCHAR(255) NOT NULL,
        SourceContext VARCHAR(100) NULL,
        TargetTable VARCHAR(100) NOT NULL,
        TargetId UNIQUEIDENTIFIER NOT NULL,
        LoadBatchId INT NOT NULL,
        CreatedDate DATETIME NOT NULL DEFAULT GETUTCDATE(),
        PRIMARY KEY (SourceSystem, SourceTable, SourceNaturalKey)
    );

Key Resolution Logic

  1. For `CoreCompany.Customer`, the `SourceNaturalKey` will be the `CUSTOMER` value from `AcquiredCompany.CUSTOMER`.
  2. For `CoreCompany.Entity`, the `SourceNaturalKey` will be a concatenation of `CUSTOMER` and `CLIENT_GROUP` (e.g., 'CUST123-A').
  3. All foreign key columns in the target (e.g., `Entity.CustomerId`) will be populated by looking up the `TargetId` in this crosswalk table.

9. Error Handling and Reject Strategy

    CREATE TABLE Migration_Reject_Log (
        RejectId BIGINT IDENTITY(1,1) PRIMARY KEY,
        BatchId INT,
        SourceSystem VARCHAR(50),
        SourceTable VARCHAR(100),
        SourceNaturalKey VARCHAR(255),
        TargetTable VARCHAR(100),
        TargetColumn VARCHAR(100),
        ErrorType VARCHAR(50),
        ErrorMessage NVARCHAR(MAX),
        SourceValue NVARCHAR(MAX),
        CreatedDate DATETIME DEFAULT GETUTCDATE()
    );

Strategy: Any record that fails validation (e.g., FK lookup failure, data type conversion error) will NOT be loaded. Instead, it will be inserted into the `Migration_Reject_Log` for manual review. The migration batch will continue, but a summary of rejects will be presented at the end.

10. Migration Execution Guide

  1. Pre-migration: Populate `CoreCompany.EntityType` and other reference tables.
  2. Step 1: Run script to populate `Migration_Key_Map` for `Customer` table.
  3. Step 2: Run ETL job to load `CoreCompany.Customer`.
  4. Step 3: Run script to populate `Migration_Key_Map` for `Entity` table.
  5. Step 4: Run ETL job to load `CoreCompany.Entity`.
  6. Step 5: Run validation SQL from the Validation Plan.
  7. Step 6: Review `Migration_Reject_Log` and `Migration_Audit_Log`.
  8. Step 7: Obtain business sign-off.

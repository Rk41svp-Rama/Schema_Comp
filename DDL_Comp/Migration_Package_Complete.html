Migration Objective

Migrate customer, entity, and document management data from Zenith legacy system (SQL Server) to a modern cloud-based platform with enhanced data governance, audit capabilities, and normalized entity-relationship structure.

Scope

  • Source System: Zenith Legacy Database (ZenithTestDDL schema)
  • Target System: Modern Cloud Platform (dbo schema)
  • Data Domain: Customer Management, Entity Hierarchy, Security, Document Management
  • Migration Type: One-time historical data migration with crosswalk-based key resolution

Key Migration Risks

⚠️ Critical Risks
  • Composite Key to GUID Migration: Source uses composite natural keys (CUSTOMER + CLIENT_GROUP), target uses surrogate GUIDs. Crosswalk table is mandatory.
  • Missing Entity Hierarchy: Source has flat CUSTOMER structure. Target has Customer → Enterprise → Entity hierarchy. Business rules needed to determine hierarchy placement.
  • PII/Tax Data: SSN, FEDERAL_ID, STATE_ID, EIN fields require encryption, masking, and compliance validation.
  • No Declared Foreign Keys in Source: Relationships must be inferred from naming conventions and business logic.
  • Document Storage Model Change: Source stores document content in DBFILE_DETAIL.DBFILE_DATA. Target uses DocumentId reference. Content migration strategy undefined.
⚠️ Medium Risks
  • CLIENT_GROUP Semantics: Source CLIENT_GROUP field purpose unclear. May map to EntityType, EntityGroup, or be deprecated.
  • Address Normalization: Source has multiple address fields (CUSTOMER_ADDR, ACCOUNT_ADDR1-4). Target uses BusinessPartyId reference. Address migration requires separate BusinessParty load.
  • Audit Field Defaults: Target has extensive audit fields (CreatedBy, CreatedDate, ModifiedBy, ModifiedDate). Source only has CHANGED/CHANGED_BY. Migration user/date strategy needed.
  • IsActive/Deactivation Logic: Target has IsActive flag and DeactivatedDate/DeactivatedBy. Source has no equivalent. Default strategy required.

Key Assumptions

  • Each source CUSTOMER record will create one target Customer record
  • Enterprise and Entity records will be generated based on CLIENT_GROUP or other business rules (SME decision required)
  • All migrated records will be marked as IsActive = 1 unless business rules specify otherwise
  • Migration will use a dedicated system user GUID for CreatedBy/ModifiedBy audit fields
  • Source CHANGED date will map to both CreatedDate and ModifiedDate in target
  • BusinessPartyId will be generated separately and linked via crosswalk
  • EntityType reference data will be pre-loaded before entity migration
  • Document content migration is out of scope for this phase (metadata only)
  • CUSTOMER_SECURITY will not be migrated (target uses different security model)
  • DOCUMENT_ID counter values will be preserved for reference but not used for target key generation

🔑 4. Crosswalk and Key Management Plan

Overview

The migration requires a comprehensive crosswalk strategy to map source natural keys (composite keys like CUSTOMER + CLIENT_GROUP) to target surrogate keys (GUIDs). This section defines the crosswalk table design, key generation rules, and resolution logic.

Crosswalk Table Design

CREATE TABLE Migration_Key_Map ( CrosswalkId UNIQUEIDENTIFIER NOT NULL DEFAULT NEWID(), SourceSystem VARCHAR(50) NOT NULL, SourceTable VARCHAR(100) NOT NULL, SourceNaturalKey VARCHAR(500) NOT NULL, SourceContext VARCHAR(100) NULL, TargetTable VARCHAR(100) NOT NULL, TargetId UNIQUEIDENTIFIER NOT NULL, LoadBatchId VARCHAR(50) NOT NULL, CreatedDate DATETIME2(7) NOT NULL DEFAULT GETDATE(), CONSTRAINT PK_Migration_Key_Map PRIMARY KEY (CrosswalkId), CONSTRAINT UQ_Migration_Key_Map UNIQUE (SourceSystem, SourceTable, SourceNaturalKey, SourceContext, TargetTable) ); CREATE INDEX IX_Migration_Key_Map_Source ON Migration_Key_Map (SourceSystem, SourceTable, SourceNaturalKey); CREATE INDEX IX_Migration_Key_Map_Target ON Migration_Key_Map (TargetTable, TargetId);

Source Natural Keys

Source Table Natural Key Components Concatenation Format Example
CUSTOMER CUSTOMER + CLIENT_GROUP RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP) 'ABC|1'
CUSTOMER_SECURITY USER_NAME + CUSTOMER RTRIM(USER_NAME) + '|' + RTRIM(CUSTOMER) 'JSMITH|ABC'
DBFILE CUSTOMER + CLIENT + DBFILE_RESOURCE + DBFILE_PATH + DBFILE_NAME + VERSION RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT) + '|' + RTRIM(DBFILE_RESOURCE) + '|' + RTRIM(DBFILE_PATH) + '|' + RTRIM(DBFILE_NAME) + '|' + CAST(VERSION AS VARCHAR) 'ABC|0001|DOC|/path/|file.pdf|1'
DBFILE_DETAIL CUSTOMER + DBFILE_ID + DBFILE_LINE RTRIM(CUSTOMER) + '|' + CAST(DBFILE_ID AS VARCHAR) + '|' + CAST(DBFILE_LINE AS VARCHAR) 'ABC|12345|1'

Target Surrogate Keys

Target Table Surrogate Key Column Generation Method Deterministic?
Customer CustomerId Deterministic GUID from CUSTOMER + CLIENT_GROUP Yes
Enterprise Id Deterministic GUID from CUSTOMER (if 1:1 mapping) Yes
Entity Id Deterministic GUID from CUSTOMER + CLIENT_GROUP Yes

Deterministic Key Generation Rules

-- Customer ID Generation DECLARE @SourceNaturalKey VARCHAR(500) = RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP); DECLARE @CustomerId UNIQUEIDENTIFIER = CAST(HASHBYTES('MD5', 'CUSTOMER|' + @SourceNaturalKey) AS UNIQUEIDENTIFIER); -- Enterprise ID Generation (1:1 with Customer) DECLARE @EnterpriseId UNIQUEIDENTIFIER = CAST(HASHBYTES('MD5', 'ENTERPRISE|' + @SourceNaturalKey) AS UNIQUEIDENTIFIER); -- Entity ID Generation DECLARE @EntityId UNIQUEIDENTIFIER = CAST(HASHBYTES('MD5', 'ENTITY|' + @SourceNaturalKey) AS UNIQUEIDENTIFIER);

Source Context Handling

SourceContext is used to avoid collisions when the same source key can feed multiple target objects. For example:

  • Context = 'CUSTOMER': CUSTOMER natural key → Customer.CustomerId
  • Context = 'ENTERPRISE': CUSTOMER natural key → Enterprise.Id
  • Context = 'ENTITY': CUSTOMER natural key → Entity.Id

Crosswalk Population Process

-- Step 1: Generate Customer IDs and populate crosswalk INSERT INTO Migration_Key_Map (SourceSystem, SourceTable, SourceNaturalKey, SourceContext, TargetTable, TargetId, LoadBatchId) SELECT 'Zenith' AS SourceSystem, 'CUSTOMER' AS SourceTable, RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP) AS SourceNaturalKey, 'CUSTOMER' AS SourceContext, 'Customer' AS TargetTable, CAST(HASHBYTES('MD5', 'CUSTOMER|' + RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP)) AS UNIQUEIDENTIFIER) AS TargetId, @BatchId AS LoadBatchId FROM ZenithTestDDL.CUSTOMER; -- Step 2: Generate Enterprise IDs and populate crosswalk INSERT INTO Migration_Key_Map (SourceSystem, SourceTable, SourceNaturalKey, SourceContext, TargetTable, TargetId, LoadBatchId) SELECT 'Zenith' AS SourceSystem, 'CUSTOMER' AS SourceTable, RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP) AS SourceNaturalKey, 'ENTERPRISE' AS SourceContext, 'Enterprise' AS TargetTable, CAST(HASHBYTES('MD5', 'ENTERPRISE|' + RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP)) AS UNIQUEIDENTIFIER) AS TargetId, @BatchId AS LoadBatchId FROM ZenithTestDDL.CUSTOMER; -- Step 3: Generate Entity IDs and populate crosswalk INSERT INTO Migration_Key_Map (SourceSystem, SourceTable, SourceNaturalKey, SourceContext, TargetTable, TargetId, LoadBatchId) SELECT 'Zenith' AS SourceSystem, 'CUSTOMER' AS SourceTable, RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP) AS SourceNaturalKey, 'ENTITY' AS SourceContext, 'Entity' AS TargetTable, CAST(HASHBYTES('MD5', 'ENTITY|' + RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP)) AS UNIQUEIDENTIFIER) AS TargetId, @BatchId AS LoadBatchId FROM ZenithTestDDL.CUSTOMER;

FK Resolution Logic

-- Resolve CustomerId for Enterprise.CustomerId FK SELECT e.Id AS EnterpriseId, xw.TargetId AS CustomerId FROM Enterprise e INNER JOIN Migration_Key_Map xw_ent ON e.Id = xw_ent.TargetId AND xw_ent.TargetTable = 'Enterprise' INNER JOIN Migration_Key_Map xw ON xw.SourceSystem = 'Zenith' AND xw.SourceTable = 'CUSTOMER' AND xw.SourceNaturalKey = xw_ent.SourceNaturalKey AND xw.SourceContext = 'CUSTOMER' AND xw.TargetTable = 'Customer'; -- Resolve EnterpriseId for Entity.EnterpriseId FK SELECT ent.Id AS EntityId, xw.TargetId AS EnterpriseId FROM Entity ent INNER JOIN Migration_Key_Map xw_ent ON ent.Id = xw_ent.TargetId AND xw_ent.TargetTable = 'Entity' INNER JOIN Migration_Key_Map xw ON xw.SourceSystem = 'Zenith' AND xw.SourceTable = 'CUSTOMER' AND xw.SourceNaturalKey = xw_ent.SourceNaturalKey AND xw.SourceContext = 'ENTERPRISE' AND xw.TargetTable = 'Enterprise';

Re-run Handling

Deterministic key generation ensures that re-running the migration produces the same GUIDs:

  • Same source natural key → Same target GUID
  • Crosswalk table uses UNIQUE constraint to prevent duplicates
  • Use MERGE or INSERT...ON CONFLICT for idempotent loads
  • LoadBatchId tracks each migration run

Duplicate Prevention

-- Check for duplicate source natural keys before migration SELECT RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP) AS SourceNaturalKey, COUNT(*) AS DuplicateCount FROM ZenithTestDDL.CUSTOMER GROUP BY RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP) HAVING COUNT(*) > 1; -- If duplicates found, investigate and resolve before migration

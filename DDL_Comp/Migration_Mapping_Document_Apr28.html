1. Executive Summary

Migration Objective

Migrate customer, entity, and document management data from the legacy Zenith system (SQL Server-based) to a modern cloud-native platform with GUID-based surrogate keys, enhanced audit capabilities, and normalized entity hierarchy.

Scope

Source System: Zenith Legacy System (ZenithTestDDL schema)
Target System: Modern Cloud Platform (dbo schema)
Migration Approach: Target-driven mapping with crosswalk-based key resolution
Data Domains: Customer Management, Entity Hierarchy, Document Management, Security

Document Management DBFILE, DBFILE_DETAIL, DOCUMENT_ID (3) Document, DocumentDetail, CustomerDocument (3)
Audit/History None EntityHistory (1)

Key Migration Risks

🚨 Critical Risks

  • Composite Key to GUID Migration: Source uses composite keys (CUSTOMER + CLIENT_GROUP). Target uses GUIDs. Requires robust crosswalk strategy.
  • Entity Hierarchy Expansion: Source has flat CUSTOMER structure. Target introduces Customer → Enterprise → Entity hierarchy. Business rules needed to determine hierarchy placement.
  • Missing Security Mapping: Source CUSTOMER_SECURITY table has no direct target equivalent. Requires external IAM/RBAC system integration.
  • Document ID Counter Migration: Source DOCUMENT_ID table stores counters. Target uses GUIDs. Counter values may need preservation for audit/reference purposes.
  • Default Values: Multiple target fields require default values (IsActive, IsCascadeDeactivation, IsZenithCustomer, etc.) with no source evidence.

Key Assumptions

📝 Working Assumptions (Require SME Validation)

  1. Each source CUSTOMER record maps to one target Customer record.
  2. Source CLIENT_GROUP can be used to derive target Enterprise structure (requires business rule definition).
  3. All migrated records default to IsActive = 1 unless deactivation logic is defined.
  4. Source CUSTOMER.CHANGED_BY can map to target CreatedBy (user GUID lookup required).
  5. Source CUSTOMER.CHANGED can map to target CreatedDate and ModifiedDate.
  6. All target GUID keys will be deterministically generated using source natural keys to support re-runs.
  7. Document encoding defaults to 'UTF-8' unless source DBFILE_ENCODING provides different value.
  8. Entity fiscal year/period fields default to NULL (requires business initialization logic).
  9. IsCascadeDeactivation defaults to 0 (no cascade) unless business rule specifies otherwise.
  10. IsZenithCustomer = 1 for all migrated customers (migration flag).

Deterministic Key Generation Rules

-- Customer
CONVERT(uniqueidentifier, HASHBYTES('MD5', 'CUSTOMER|' + RTRIM(CUSTOMER)))

-- Enterprise
CONVERT(uniqueidentifier, HASHBYTES('MD5', 'ENTERPRISE|' + RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP)))

-- Entity
CONVERT(uniqueidentifier, HASHBYTES('MD5', 'ENTITY|' + RTRIM(CUSTOMER) + '|' + RTRIM(CLIENT_GROUP)))

-- Document
CONVERT(uniqueidentifier, HASHBYTES('MD5', 'DOCUMENT|' + RTRIM(CUSTOMER) + '|' + CAST(DBFILE_ID AS VARCHAR)))

-- DocumentDetail
CONVERT(uniqueidentifier, HASHBYTES('MD5', 'DOCDETAIL|' + RTRIM(CUSTOMER) + '|' + CAST(DBFILE_ID AS VARCHAR) + '|' + CAST(DBFILE_LINE AS VARCHAR)))

Re-run Handling

  • Deterministic key generation ensures same source record always generates same target GUID
  • Crosswalk table uses UNIQUE constraint to prevent duplicate mappings
  • Migration process should use MERGE or INSERT...WHERE NOT EXISTS to avoid duplicates
  • LoadBatchId tracks which batch created each mapping
  • Re-runs should use same LoadBatchId for idempotency

FK Resolution Logic

-- Example: Resolve EnterpriseId for Entity load
SELECT 
    e.Id AS EntityId,
    mkm.TargetId AS EnterpriseId
FROM Entity e
INNER JOIN Migration_Key_Map mkm
    ON mkm.SourceSystem = 'Zenith'
    AND mkm.SourceTable = 'CUSTOMER'
    AND mkm.SourceNaturalKey = e.EntityNumber  -- Assuming EntityNumber = CUSTOMER + '-' + CLIENT_GROUP
    AND mkm.SourceContext = 'ENTERPRISE'
    AND mkm.TargetTable = 'Enterprise'
WHERE e.EnterpriseId IS NULL;

Collision Prevention

  • Use SourceContext field to differentiate same source key feeding multiple target tables
  • Example: CUSTOMER + CLIENT_GROUP feeds both Enterprise (context='ENTERPRISE') and Entity (context='ENTITY')
  • Unique constraint on (SourceSystem, SourceTable, SourceNaturalKey, SourceContext, TargetTable) prevents collisions
  • Prefix in hash function ('CUSTOMER|', 'ENTERPRISE|', 'ENTITY|') provides additional collision protection

Error Categories and Handling

Error Category Description Handling Strategy Severity
MandatoryNull Required field is NULL or empty Log reject, skip record, notify SME for data correction Critical
LookupFailed FK lookup in crosswalk or reference table failed Log reject, skip record, verify crosswalk completeness Critical
DataTypeError Source value cannot be converted to target data type Log reject, skip record, investigate source data quality High

AmountBalanceError Amount totals do not balance Log reject, investigate scale/precision issues High

Retry Strategy

  • Transient Errors: Retry up to 3 times with exponential backoff (network, timeout)
  • Data Errors: No automatic retry - requires data correction and manual re-run
  • FK Violations: Verify load sequence, re-run dependent load groups if needed
  • Lookup Failures: Verify crosswalk completeness, re-run crosswalk population if needed

Partial Load Handling

  • Each load group is atomic - either all records succeed or entire group is rolled back
  • Use database transactions to ensure atomicity
  • If load group fails, log error, rollback transaction, fix issue, and re-run
  • Crosswalk updates are committed separately to preserve mappings even if target load fails

10. Migration Execution Guide

Step-by-Step Execution Plan

  1. Profile Source Data
    • Run data profiling queries on source tables
    • Identify data quality issues (NULLs, invalid values, duplicates)
    • Confirm source natural keys are unique and complete
    • Analyze value distributions for status codes, flags, and categories
    • Document findings in data quality report
  2. Confirm Source Natural Keys
    • Verify CUSTOMER is unique and non-NULL
    • Verify CUSTOMER + CLIENT_GROUP is unique
    • Verify DBFILE_ID is unique within CUSTOMER
    • Verify DBFILE_ID + DBFILE_LINE is unique within CUSTOMER
  3. Confirm Source Filter Criteria
    • Define active customer filter (if any)
    • Define date range filter (if any)
    • Exclude test/demo data (if applicable)
    • Document filter criteria in migration plan
  4. Confirm Amount Scale and Date Semantics
    • Verify KILOBYTES represents actual kilobytes (not bytes)
    • Confirm CHANGED and CREATED are in correct timezone
    • Validate date ranges are reasonable
  5. Load Reference/Static Tables (Load Group 0)
    • Seed EntityType table with valid entity types
    • Seed CustomerType table with valid customer types
    • Seed DocumentType table with valid document types
    • Verify reference data completeness
  6. Generate Deterministic Target Keys (Load Group 1)
    • Create Migration_Key_Map table
    • Create User crosswalk (map source user codes to target User GUIDs)
    • Verify crosswalk structure and indexes
  7. Populate Crosswalk Table (Load Group 1)
    • Generate Customer crosswalk entries
    • Generate Enterprise crosswalk entries
    • Generate Entity crosswalk entries
    • Generate Document crosswalk entries
    • Verify crosswalk completeness
  8. Load Customer (Load Group 2)
    • Extract distinct CUSTOMER records from source
    • Transform per mapping rules
    • Load into Customer table
    • Update Customer crosswalk
    • Run validation SQL
  9. Load Enterprise (Load Group 3)
    • Extract distinct CUSTOMER + CLIENT_GROUP combinations
    • Transform per mapping rules
    • Resolve CustomerId FK via crosswalk
    • Load into Enterprise table
    • Update Enterprise crosswalk
    • Run validation SQL
  10. Load Entity (Load Group 4)
    • Extract CUSTOMER records
    • Transform per mapping rules
    • Resolve EnterpriseId and CustomerId FKs via crosswalk
    • Load into Entity table
    • Update Entity crosswalk
    • Run validation SQL

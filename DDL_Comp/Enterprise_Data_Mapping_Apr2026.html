1. Executive Summary

Migration Objective

Migrate customer, entity, and document management data from the legacy AcquiredCompany system to the modern CoreCompany platform. This migration supports the enterprise consolidation initiative and enables unified customer and entity management across the organization.

Scope

  • Source System: AcquiredCompany (Legacy SQL Server Database)
  • Target System: CoreCompany (Modern Multi-Tenant Platform)
  • Migration Type: One-time data migration with full historical preservation
  • Data Volume: To be determined during profiling phase

Key Migration Risks

  • HIGH: No direct source for Enterprise table - requires business logic to determine enterprise grouping from legacy CLIENT_GROUP
  • HIGH: No direct source for Entity table - requires determination of whether legacy CUSTOMER records represent enterprises or entities
  • HIGH: Target uses GUID-based surrogate keys while source uses character-based natural keys - requires comprehensive crosswalk strategy
  • MEDIUM: Document management structure differs significantly - DBFILE/DBFILE_DETAIL must be reconstructed into target document repository
  • MEDIUM: User security model transformation - CUSTOMER_SECURITY maps to UserCustomerMapping but user identity resolution is unclear
  • MEDIUM: Multiple target audit/history tables have no source - migration metadata must be generated

Key Assumptions

  • Legacy CUSTOMER table represents what will become Customer in target (not Entity or Enterprise)
  • CLIENT_GROUP field may indicate enterprise-level grouping but requires business validation
  • All migrated records will be marked as IsActive = 1 unless business rules specify otherwise
  • Tax identifiers (SSN, FEDERAL_ID, STATE_ID) map to target TIN field with business rule for precedence
  • User identity resolution will be handled through separate user migration or identity mapping process
  • Document content from DBFILE_DETAIL will be reconstructed and stored in target document repository
  • All audit fields (CreatedBy, CreatedDate, ModifiedBy, ModifiedDate) will use migration service account and migration timestamp

Document Migration Strategy

Document Repository Migration: The source system stores documents in DBFILE (metadata) and DBFILE_DETAIL (content) tables. The target system likely uses a document repository (file system, blob storage, or document management system). Migration strategy must address:
  • Document Reconstruction: Reassemble document content from DBFILE_DETAIL.DBFILE_DATA (char 1024) lines ordered by DBFILE_LINE
  • Document Storage: Store reconstructed documents in target document repository
  • Document Metadata: Create Document table records with metadata from DBFILE (name, path, encoding, size, created date)
  • Document Linking: Populate CustomerDocument table to link documents to customers
  • Document ID Generation: Generate new DocumentId (GUID) for each document and store in crosswalk

Table-Level Assumptions

  • Documents must be reconstructed from DBFILE_DETAIL lines and stored in target document repository

Re-Run Handling

  • Use deterministic GUID generation (hash-based) to ensure same source record always generates same target ID
  • Check crosswalk before inserting - if mapping exists, reuse existing target ID
  • Use MERGE statements for idempotent loads
  • Tag each crosswalk entry with LoadBatchId for traceability

Duplicate Prevention

  • Enforce unique constraint on crosswalk: (SourceSystem, SourceTable, SourceNaturalKey, SourceContext, TargetTable)
  • Validate source natural key uniqueness before migration
  • Use MERGE with WHEN MATCHED / WHEN NOT MATCHED logic

FK Resolution Logic

-- Resolve foreign key from crosswalk
SELECT 
    target.Id,
    target.EntityName,
    xwalk_customer.TargetId AS CustomerId,
    xwalk_enterprise.TargetId AS EnterpriseId
FROM source.CUSTOMER src
INNER JOIN Migration_Key_Map xwalk_customer
    ON xwalk_customer.SourceSystem = 'AcquiredCompany'
    AND xwalk_customer.SourceTable = 'CUSTOMER'
    AND xwalk_customer.SourceNaturalKey = RTRIM(LTRIM(src.CUSTOMER)) + '|' + RTRIM(LTRIM(src.CLIENT_GROUP))
    AND xwalk_customer.TargetTable = 'Customer'
LEFT JOIN Migration_Key_Map xwalk_enterprise
    ON xwalk_enterprise.SourceSystem = 'AcquiredCompany'
    AND xwalk_enterprise.SourceTable = 'CUSTOMER'
    AND xwalk_enterprise.SourceNaturalKey = RTRIM(LTRIM(src.CLIENT_GROUP))
    AND xwalk_enterprise.TargetTable = 'Enterprise'

Collision Prevention for Shared Keys

  • Use SourceContext field to differentiate when same source key feeds multiple target objects
  • Example: If CUSTOMER code is used for both Customer and Entity, use SourceContext = 'Customer' vs 'Entity'
  • Enforce unique constraint includes SourceContext to prevent collisions

10. Migration Execution Guide

Pre-Migration Phase

  1. Profile Source Data
    • Run data profiling queries on source tables
    • Validate source natural key uniqueness
    • Identify data quality issues (nulls, duplicates, invalid values)
    • Confirm source record counts
  2. Confirm Source Natural Keys
    • Validate CUSTOMER + CLIENT_GROUP uniqueness
    • Validate USER_NAME + CUSTOMER uniqueness
    • Validate DBFILE_ID uniqueness within CUSTOMER
  3. Confirm Source Filter Criteria
    • Define filter criteria for CUSTOMER_SECURITY (CURRENT_FLAG = '1'?)
    • Define filter criteria for DBFILE (all records or subset?)
    • Confirm no test/dummy records should be migrated
  4. Resolve SME Decisions
    • Customer vs Entity vs Enterprise classification
    • CLIENT_GROUP to Enterprise mapping logic
    • Tax identifier precedence rule
    • User identity resolution strategy
    • Default CustomerType assignment
    • Default EntityType assignment
    • Default ChartOfAccountsName assignment
    • Default PreferredCurrency assignment
    • Document migration strategy
  5. Pre-Populate Reference Tables
    • Load CustomerType reference data
    • Load EntityType reference data
    • Confirm default values exist
  6. Create Migration Infrastructure
    • Create Migration_Key_Map table
    • Create Migration_Audit_Log table
    • Create Migration_Reject_Log table
    • Create migration service account user

Migration Execution Phase

  1. Generate Migration Batch ID
    • Create new BatchId (GUID) for this migration run
    • Record batch start time
  2. Load Group 0: Reference Tables
    • Validate CustomerType and EntityType are pre-populated
  3. Load Group 1: Crosswalk Preparation
    • Generate deterministic Customer IDs from source CUSTOMER + CLIENT_GROUP
    • Populate Migration_Key_Map for Customer
    • Validate no duplicate natural keys
  4. Load Group 1: Customer
    • Extract source CUSTOMER records
    • Transform and map to target Customer table
    • Lookup CustomerId from crosswalk
    • Validate mandatory fields
    • Insert into Customer table
    • Log audit entry
  5. Load Group 1: Enterprise (if applicable)
    • Generate deterministic Enterprise IDs based on CLIENT_GROUP grouping logic
    • Populate Migration_Key_Map for Enterprise
    • Extract and transform source data
    • Insert into Enterprise table
    • Log audit entry
  6. Load Group 1: Entity (if applicable)
    • Generate deterministic Entity IDs
    • Populate Migration_Key_Map for Entity
    • Extract and transform source data
    • Lookup foreign keys from crosswalk
    • Insert into Entity table
    • Log audit entry
  7. Load Group 1: User Identity Resolution
    • Extract distinct USER_NAME values from CUSTOMER_SECURITY
    • Resolve to target UserId (GUID)
    • Populate Migration_Key_Map for User
    • Log unresolved users to reject table
  8. Load Group 1: UserCustomerMapping
    • Extract source CUSTOMER_SECURITY records
    • Lookup UserId and CustomerId from crosswalk
    • Transform CURRENT_FLAG to IsActive
    • Insert into UserCustomerMapping table
    • Log audit entry
  9. Load Group 1: Document Migration
    • Extract DBFILE and DBFILE_DETAIL records
    • Reconstruct document content from DBFILE_DETAIL lines
    • Store documents in target document repository
    • Generate DocumentId for each document
    • Populate Migration_Key_Map for Document
    • Log audit entry
  10. Load Group 1: CustomerDocument
    • Extract source DBFILE records
    • Lookup CustomerId and DocumentId from crosswalk
    • Insert into CustomerDocument table
    • Log audit entry
  11. Load Group 10: History Tables (optional)
    • Insert migration audit records into CustomerHistory and EntityHistory
    • Document initial migration event

Post-Migration Phase

  1. Run Validation SQL
    • Execute all validation queries from section 6
    • Review results and identify issues
  2. Review Rejects
    • Query Migration_Reject_Log for rejected records
    • Analyze error types and counts
    • Determine root cause for rejects
    • Plan remediation
  3. Reconcile Counts and Totals
    • Compare source and target row counts
    • Validate control totals by CLIENT_GROUP/OfficeCode
    • Confirm crosswalk completeness
  4. Resolve SME Decisions
    • Present SME decision tracker to business
    • Obtain decisions on blocking items
    • Update migration logic based on decisions
  5. Obtain Business Sign-Off
    • Present migration results to business stakeholders
    • Review validation results
    • Review reject log
    • Obtain formal sign-off
  6. Freeze Migration Batch
    • Mark migration batch as complete in audit log
    • Archive migration logs and crosswalk
    • Document final migration statistics
  7. Prepare Cutover
    • Plan production cutover
    • Define rollback strategy
    • Prepare cutover checklist
    • Schedule cutover window

Key Migration Risks

HIGH RISK:
  • Surrogate Key Generation: Target uses UNIQUEIDENTIFIER (GUID) keys while source uses composite natural keys. Crosswalk table is mandatory.
  • Missing Foreign Key Relationships: Source DDL does not declare FK constraints. Relationships must be inferred and validated.
  • BusinessParty Creation Logic: Target requires BusinessPartyId for Customer and Entity, but source has no equivalent. Creation strategy needs SME approval.
  • Entity Type Classification: Target requires EntityTypeId, but source has no entity type classification. Default mapping strategy required.
MEDIUM RISK:
  • Address Normalization: Source has multiple address fields (CUSTOMER_ADDR, CUSTOMER_ADDR2, ACCOUNT_ADDR1-4). Target has single EntityAddress field. Concatenation logic needs validation.
  • Audit Field Population: Target has extensive audit fields (CreatedBy, CreatedDate, ModifiedBy, ModifiedDate). Source only has CHANGED and CHANGED_BY. Migration user/date strategy required.
  • Active Status Defaults: Target IsActive fields need default values. Business rules for determining active status required.
  • Document ID Mapping: Source DOCUMENT_ID table appears to be a counter/sequence table. Relationship to target DocumentId field unclear.

Key Assumptions

  • Source CUSTOMER table represents both Customer and Entity concepts in target model
  • CLIENT_GROUP field in source maps to Enterprise concept in target
  • All migrated records will be marked as IsActive = 1 unless SME provides deactivation criteria
  • Migration will use a system user GUID for all CreatedBy/ModifiedBy audit fields
  • Source CHANGED date will map to both CreatedDate and ModifiedDate in target
  • CUSTOMER_SECURITY table provides user access control but does not directly map to target schema
  • DBFILE and DBFILE_DETAIL tables represent document storage but require separate document management migration strategy

Key SME Decisions Required

BLOCKING DECISIONS - Migration cannot proceed without these:
  1. BusinessParty Strategy: How should BusinessParty records be created? One per Customer? Shared across Customers?
  2. Enterprise Hierarchy: Should CLIENT_GROUP create Enterprise records? What is the hierarchy relationship?
  3. Entity Type Mapping: What EntityType should be assigned to migrated entities? Default type or derived from source data?
  4. Customer Type Classification: Target requires CustomerTypeId. How should source customers be classified?
  5. Active Status Logic: What criteria determine if a migrated record should be active or inactive?
  6. Address Concatenation Rules: How should multiple source address fields be combined into single target field?
  7. Document Migration Scope: Should DBFILE/DBFILE_DETAIL be migrated? If yes, what is the target document storage strategy?
  8. User Security Migration: Should CUSTOMER_SECURITY data be migrated? If yes, to which target table/system?
  9. Deactivation Handling: Should any historical records be marked as deactivated? What are the criteria?
  10. Currency and Fiscal Settings: What default values for PreferredCurrencyId, CurrentFiscalYear, etc.?
  11. Chart of Accounts Assignment: How should ChartOfAccountsNameId be assigned to entities?
  12. Office Code Assignment: Target Customer requires OfficeCode. What is the source or default value?

3. Transformation Rules Library

3.1 String Trimming and Cleaning

-- Rule: TRIM_CHAR_FIELD -- Purpose: Remove leading/trailing spaces from CHAR fields -- Usage: Apply to all CHAR source fields RTRIM(LTRIM(source_field)) -- Rule: CLEAN_STRING -- Purpose: Remove leading/trailing spaces and handle NULL -- Usage: Apply to all string fields CASE WHEN RTRIM(LTRIM(source_field)) = '' THEN NULL ELSE RTRIM(LTRIM(source_field)) END

3.2 Data Type Conversion

-- Rule: CHAR_TO_VARCHAR -- Purpose: Convert CHAR to VARCHAR with trimming -- Usage: Apply to all CHAR to VARCHAR mappings CAST(RTRIM(LTRIM(source_field)) AS VARCHAR(length)) -- Rule: CHAR_TO_NVARCHAR -- Purpose: Convert CHAR to NVARCHAR with trimming and Unicode support -- Usage: Apply to all CHAR to NVARCHAR mappings CAST(RTRIM(LTRIM(source_field)) AS NVARCHAR(length)) -- Rule: CHAR_TO_BIT -- Purpose: Convert CHAR flag to BIT -- Usage: Apply to flag/boolean conversions CASE WHEN RTRIM(LTRIM(source_field)) IN ('Y', '1', 'T', 'TRUE') THEN 1 WHEN RTRIM(LTRIM(source_field)) IN ('N', '0', 'F', 'FALSE') THEN 0 ELSE NULL END

3.3 Date Handling

-- Rule: DATETIME_TO_DATETIME2 -- Purpose: Convert DATETIME to DATETIME2 -- Usage: Apply to all date field conversions CAST(source_date AS DATETIME2(7)) -- Rule: DEFAULT_MIGRATION_DATE -- Purpose: Use migration date when source date is NULL -- Usage: Apply to audit date fields COALESCE(source_date, GETDATE()) -- Rule: PRESERVE_SOURCE_DATE -- Purpose: Preserve source date or use NULL -- Usage: Apply to optional date fields source_date

3.4 Surrogate Key Generation

-- Rule: DETERMINISTIC_GUID_GENERATION -- Purpose: Generate deterministic GUID from source natural key -- Usage: Apply to all surrogate key generation -- Note: Ensures re-runability and consistency CAST( HASHBYTES('MD5', CONCAT( 'TableName:', RTRIM(LTRIM(source_key1)), '|', RTRIM(LTRIM(source_key2)) ) ) AS UNIQUEIDENTIFIER ) -- Rule: RANDOM_GUID_GENERATION -- Purpose: Generate random GUID (use only if deterministic not required) -- Usage: Apply to non-critical surrogate keys NEWID()

3.5 Lookup Resolution via Crosswalk

4.8 Re-run Handling

Re-runability Strategy:
  • Use deterministic GUID generation to ensure same source key always generates same target GUID
  • Crosswalk table has unique constraint on (SourceSystem, SourceTable, SourceNaturalKey, SourceContext, TargetTable)
  • Use MERGE statements instead of INSERT to handle re-runs
  • Track LoadBatchId to identify which batch created each mapping
  • Implement idempotent load logic that can safely re-run without duplicates

8.2 Audit Log Usage Pattern

-- Start of migration step DECLARE @AuditLogId UNIQUEIDENTIFIER = NEWID(); DECLARE @StartTime DATETIME2(7) = GETDATE(); INSERT INTO Migration_Audit_Log ( AuditLogId, BatchId, SourceSystem, SourceTable, TargetTable, LoadGroup, StartTime, Status, ExecutedBy ) VALUES ( @AuditLogId, @BatchId, 'Zenith', 'CUSTOMER', 'Customer', 3, @StartTime, 'Running', SYSTEM_USER ); -- Execute migration step BEGIN TRY -- Migration logic here DECLARE @RecordsRead INT = (SELECT COUNT(*) FROM ZenithTestDDL.CUSTOMER WHERE CUSTOMER_NAME IS NOT NULL); DECLARE @RecordsInserted INT = @@ROWCOUNT; -- Update audit log on success UPDATE Migration_Audit_Log SET RecordsRead = @RecordsRead, RecordsInserted = @RecordsInserted, RecordsRejected = 0, EndTime = GETDATE(), Status = 'Completed' WHERE AuditLogId = @AuditLogId; END TRY BEGIN CATCH -- Update audit log on failure UPDATE Migration_Audit_Log SET EndTime = GETDATE(), Status = 'Failed', ErrorMessage = ERROR_MESSAGE() WHERE AuditLogId = @AuditLogId; THROW; END CATCH;

8.3 Audit Log Reporting Queries

-- Migration summary by batch SELECT BatchId, COUNT(*) as TotalSteps, SUM(RecordsRead) as TotalRecordsRead, SUM(RecordsInserted) as TotalRecordsInserted, SUM(RecordsRejected) as TotalRecordsRejected, MIN(StartTime) as MigrationStartTime, MAX(EndTime) as MigrationEndTime, SUM(DurationSeconds) as TotalDurationSeconds, SUM(CASE WHEN Status = 'Completed' THEN 1 ELSE 0 END) as CompletedSteps, SUM(CASE WHEN Status = 'Failed' THEN 1 ELSE 0 END) as FailedSteps FROM Migration_Audit_Log GROUP BY BatchId ORDER BY MIN(StartTime) DESC; -- Failed migration steps SELECT BatchId, SourceTable, TargetTable, StartTime, EndTime, ErrorMessage FROM Migration_Audit_Log WHERE Status = 'Failed' ORDER BY StartTime DESC; -- Migration performance by table SELECT TargetTable, COUNT(*) as MigrationRuns, AVG(RecordsRead) as AvgRecordsRead, AVG(RecordsInserted) as AvgRecordsInserted, AVG(DurationSeconds) as AvgDurationSeconds, MAX(DurationSeconds) as MaxDurationSeconds FROM Migration_Audit_Log WHERE Status = 'Completed' GROUP BY TargetTable ORDER BY AvgDurationSeconds DESC;
